hypha-browser-use

Drive the user's real Chrome stable on macOS via Hypha RPC. No detection battles, no emulator, no Web Store publishing. Two background processes, ~40 tools, one paste-to-Claude prompt.

✓ working in real Chrome 148 ✓ 28/28 tools verified MIT licensed

What it does

Runs an MV3 Chrome extension inside your real Chrome (loaded via the official Extensions.loadUnpacked CDP API — no --load-extension, no Web Store) and exposes a Hypha RPC service with tools for tabs, navigation, DOM manipulation, screenshots, JS evaluation, cookies, file uploads/downloads, and more. An AI agent (or any HTTP client) drives the browser through ordinary HTTPS POSTs.

Use it for: autonomous logins, email automation, scraping, file uploads/downloads, institutional IT systems (Outlook/SharePoint/Teams web), banking, anything browser-based. For 2FA push prompts (e.g. KTH biometric MS Authenticator), the agent pauses and notifies you to tap "Approve" on your real phone — no emulator, no biometric bypass.

One-paste setup

Clone the repo, set 3 env vars, run one command:

git clone https://github.com/oeway/hypha-browser-use
cd hypha-browser-use
export HYPHA_TOKEN=...        # get from https://hypha.aicell.io
export HYPHA_WORKSPACE=ws-user-...
export HYPHA_SERVER_URL=https://hypha.aicell.io
./scripts/run.sh --headless

This launches Chrome with the extension and registers a service in your Hypha workspace. The script prints a service URL — paste it (and this docs URL) into a Claude session.

2FA / biometric handoff

The extension cannot bypass biometric (and shouldn't try). For push 2FA, the agent reads the displayed match number from the browser, calls notify_user, and waits for the page to advance:

// agent-side pseudocode
await call("navigate",        {url: "https://login.microsoftonline.com"});
await call("fill",            {selector: "input[type=email]", value: "you@org.edu"});
await call("click",           {selector: "input[type=submit]"});
await call("wait_for_selector", {selector: ".display-sign-in-number", timeout_ms: 15000});
const num = await call("read_text", {selector: ".display-sign-in-number"});
await call("notify_user",     {message: `Approve sign-in: tap ${num} on your phone`});
await call("wait_for_selector", {selector: ".inbox-ready", timeout_ms: 60000});  // user taps

Architecture

Claude → HTTPS → Hypha cloud → SSE → MV3 SW (in real Chrome) → chrome.* APIs

The Node supervisor holds Chrome alive via --remote-debugging-pipe and installs
the extension via the official Extensions.loadUnpacked CDP method. The SW
registers a Hypha service using the HTTP-streaming transport (WebSocket
hangs in MV3 SW context).

Why this approach

  • Real user fingerprint. It's the user's real Chrome — Cloudflare, Akamai, banks, Google login all see a normal user. No detection battle.
  • No Web Store publishing. The extension is loaded as unpacked dev via CDP. Zero distribution friction.
  • No sudo / no managed policy. Chrome stable in 2026 refuses --load-extension, but the CDP Extensions.loadUnpacked path works without admin.
  • 2FA is solved out-of-band. The user taps on their real phone. Works for biometric/MFA where every emulator fails.
  • Clean lifecycle. Node supervisor + Chrome share a process tether; one ./scripts/stop.sh shuts both down.
